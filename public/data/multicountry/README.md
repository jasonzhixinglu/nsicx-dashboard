# multicountry-nsicx dashboard data export

This directory is the data feed for the multi-country NSICX dashboard.
It is regenerated by `scripts/build_dashboard_export.py` from the
existing pipeline artifacts (production DNS estimates, Consensus surveys,
Haver CPI, anchoring regressions). All values are pre-computed; the
dashboard does no estimation.

## Layout

```
output/dashboard/
  README.md                        (this file)
  manifest.json                    machine-readable index
  schema_version.txt               bare integer
  countries/
    {slug}/
      states.json                  filtered + smoothed L/S/C with SEs, plus lambda
      cpi.json                     CPI YoY series (full sample)
      surveys.json                 Consensus survey rows for the snapshot vintages
      mle.json                     MLE summary (params, log-lik, T, etc.)
  cross_country/
    anchoring.json                 anchoring regressions, both specs
```

`{slug}` is the lowercase, underscored country name (`usa`, `south_korea`,
`uk`, etc.) — same convention used in `output/dns_production/`.

## Quick start

**Python:**

```python
import json
from pathlib import Path

ROOT = Path("output/dashboard")
manifest = json.load(open(ROOT / "manifest.json"))
print(manifest["schema_version"], len(manifest["countries"]))

usa_states = json.load(open(ROOT / "countries" / "usa" / "states.json"))
print(usa_states["lambda"], len(usa_states["filtered"]))
```

**JavaScript / browser:**

```js
const manifest = await fetch("/dashboard/manifest.json").then(r => r.json());
const usa = await fetch("/dashboard/countries/usa/states.json").then(r => r.json());
console.log(usa.lambda, usa.filtered.length);
```

## File schemas

### `manifest.json`

```json
{
  "schema_version": 1,
  "generated_at":   "2026-05-04T...",
  "survey_periods": ["2026-01", "2026-02", "2026-03", "2026-04"],
  "countries": [
    {"slug": "usa", "name": "USA", "last_vintage": "2026-04",
     "T": 304, "n_surveys": 8},
    ...
  ]
}
```

- `schema_version`: bumped on any breaking change to file shapes/fields.
  Consumers should refuse to render unknown major versions.
- `generated_at`: UTC ISO-8601 timestamp of the build run.
- `survey_periods`: list of survey-month strings exported under each
  country's `surveys.json`.
- `countries[*].T`: number of monthly state observations.
- `countries[*].n_surveys`: number of survey rows exported (counts both
  ST and LT sources where available).

### `countries/{slug}/states.json`

```json
{
  "country": "USA",
  "lambda":  0.0991,
  "filtered": [
    {"d": "2001-01", "L": 2.6472, "S": -0.3613, "C": 0.1142,
     "L_se": 0.4927, "S_se": 1.3372, "C_se": 1.3957},
    ...
  ],
  "smoothed": [...]
}
```

- `d`: month string `"YYYY-MM"`.
- `L`, `S`, `C`: Nelson-Siegel level, slope, curvature factors at the
  month, in **percent**.
- `L_se`, `S_se`, `C_se`: standard errors from the Kalman state
  covariance diagonal (filtering for `filtered`, smoothing for
  `smoothed`). 68% CI: `± SE`. 95% CI: `± 1.96 SE`.
  **Caveat**: these reflect filtering/data uncertainty only; parameter-
  estimation uncertainty (e.g. on lambda, q_L, q_S, q_C, v_1..v_6) is
  not included. For that you'd need a parametric or block bootstrap.
- `lambda`: NS curvature parameter (decay rate). Required to evaluate
  the curve at any horizon. Used by both filtered and smoothed factors.

NS curve evaluations from the factors:

```
avg annualized rate over 0..h months:
  pi(h) = L + A(h) * S + (A(h) - exp(-lambda*h)) * C
  where A(h) = (1 - exp(-lambda*h)) / (lambda * h)

instantaneous forward at horizon h:
  f(h) = L + exp(-lambda*h) * S + lambda*h*exp(-lambda*h) * C
```

### `countries/{slug}/cpi.json`

```json
{
  "country": "USA",
  "yoy_pct": [
    {"d": "1991-01", "v": 5.7138},
    ...
  ]
}
```

- `yoy_pct[*].v`: realized YoY CPI inflation in percent, computed as
  `(log(SA_CPI_t) - log(SA_CPI_{t-12})) * 100` from the X-12-default
  seasonally adjusted monthly CPI level (`data/processed/cpi_sa.parquet`).
  This is the same SA series used by the production strip, so the
  dashboard's displayed CPI line is internally consistent with the
  realized YTD that the model strips out of the surveys. Coverage
  starts at 2001-01 (one year after the SA build's start of 2000-01,
  which is the earliest period at which a 12-month YoY can be formed).

### `countries/{slug}/surveys.json`

```json
{
  "country": "USA",
  "survey_periods": ["2026-01", "2026-02", "2026-03", "2026-04"],
  "rows": [
    {
      "survey_period": "2026-01",
      "survey_date":   "2026-01-13",
      "source":        "ST",
      "t1": 2.7, "t2": 2.5, "t3": null, ..., "t25": null
    },
    ...
  ]
}
```

- `source`: `"ST"` (short-term, monthly Consensus) or `"LT"` (long-term,
  semi-annual Consensus).
- `t1` .. `t25`: Consensus forecast horizons. Interpretation depends on
  the survey source — for monthly ST surveys, `t1` is typically the
  current calendar year and `t2` is the next; for the semi-annual LT
  survey, the trailing horizons (`t8`..`t10`) extend further out. Some
  cells are `null` when the survey did not include that horizon.
- The dashboard interprets per-source. For the snapshot view, the
  recommended convention is "current calendar year" = `t1` from ST.
- Multiple rows per `survey_period` are possible when both ST and LT
  surveys fell in the same month.

### `countries/{slug}/mle.json`

Verbatim copy of `output/dns_production/{slug}/mle_summary.json`. Top-
level fields:

- `country`, `start`, `nobs`, `loglik`
- `lam`: NS lambda
- `params`: estimated parameters (`qL`, `qS`, `qC`, `lam`, `v1`..`v6`)
- `bse`: standard errors on the parameters (asymptotic, from MLE Hessian)

Note that `bse` differs from the `*_se` fields in `states.json`: `bse`
is parameter-estimation uncertainty; `states.json *_se` is state
filtering uncertainty.

### `cross_country/anchoring.json`

```json
{
  "main":          {"USA": {"pi_qoq": {...}, ...}, ...},
  "raw_revisions": {"USA": {"pi_qoq": {...}, ...}, ...},
  "spec_legend":   {...}
}
```

Two specs, both contain per-country regression statistics for four
RHS variables:

- `pi_qoq`: QoQ inflation surprise (annualized)
- `pi_yoy`: YoY inflation surprise
- `pi_sur`: Survey-based inflation surprise (current vs prior survey)
- `delta_ST` (main only) / `st_rev` (raw_revisions only): short-horizon
  survey revision

Each entry has `alpha`, `beta`, `p_beta`, `r2`, `T`. These are the
slope/intercept/p-value/R²/sample-size of the anchoring regression of
long-horizon forward-rate change on the surprise. The two specs differ
in how the long-horizon revision is constructed: `main` uses the DNS
filter-implied long-end change; `raw_revisions` uses the raw long-end
forecast change in the surveys.

`spec_legend` is a small dictionary of human-readable descriptions
matching the keys above.

## Conventions

- All dates are `"YYYY-MM"` strings unless explicitly a date (e.g.
  `survey_date` is `"YYYY-MM-DD"`).
- All percent values are in **percent** units (e.g. `2.5` means 2.5%).
- Slugs match `output/dns_production/{slug}/` — lowercase with
  underscores: `south_korea`, `uk`, `usa`, etc.
- Missing values are `null` (never `"NaN"` strings or NaN floats).
- Numeric values are rounded to 4 decimal places to keep file sizes
  small. The underlying parquets carry full precision.

## Schema version

| version | notes |
|---|---|
| 1     | initial; SEs are filtering-only, anchoring covers both specs |

A new major version should be cut on any breaking change to file
shapes or field names. Additive changes (new optional fields, new
files) do not require a version bump but should be noted here.
